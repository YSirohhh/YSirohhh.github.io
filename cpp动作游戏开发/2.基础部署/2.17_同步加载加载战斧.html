在游戏开发中，加载（Loading） 本质上是把数据从硬盘（慢速）搬运到内存（快速）的过程。

同步（Synchronous） 和 异步（Asynchronous） 的区别，通俗点说，就是 “死等” 和 “边走边等” 的区别。

1. 同步加载 (Synchronous Loading)

当你调用同步加载函数（比如 UE 中的 LoadObject<T>()）时，程序会停下手里所有的活儿，全身心投入到搬运工作中。

  • 特点：代码按顺序执行。加载不完，下一行代码永远不会运行。
  • 代价：如果资源很大（比如一个 4K 贴图或复杂的模型），主线程就会卡住。反映到游戏里，就是画面瞬间冻结（掉帧/卡顿）。
  • 适用场景：
  • 游戏刚启动时的 Logo 界面。
  • 关卡切换时的 Loading 进度条。
  • 非常小、能瞬间加载完的基础配置。

比喻：你去餐厅点餐，前台告诉你：“你的菜没做好之前，我不许你去座位上坐着，你必须在柜台前盯着我看。” 于是后面排队的人（其他程序逻辑）也都卡住了。

2. 异步加载 (Asynchronous Loading)

当你调用异步加载（比如 UE 中的 FStreamableManager::RequestAsyncLoad）时，你会开启一个“后台任务”。

  • 特点：主线程发出指令：“帮我搬这个货，搬好了发个微信（回调函数）通知我。” 然后主线程立刻去干别的事（比如渲染画面、处理玩家输入）。
  • 代价：逻辑变得复杂。你不能立刻使用那个资源，必须写一套“等它加载完再执行”的逻辑（Callback/Delegate）。
  • 适用场景：
  • 开放世界流送（Streaming）：当你跑向地图另一端时，远处的建筑在后台静悄悄地加载，玩家毫无察觉。
  • 技能特效预加载：角色放技能前，异步加载特效资源，保证出招时不卡顿。

3. 核心对比表

特性 同步加载 (Sync) 异步加载 (Async)
主线程状态 阻塞（卡住不动） 正常运行（丝滑顺畅）
代码复杂度 简单（直接拿来用） 复杂（需要处理回调/委托）
用户体验 可能会造成掉帧或卡顿 极致流畅，无感加载
常用函数 (UE) StaticLoadObject / LoadObject LoadPackageAsync / AsyncLoad

4. 在你的 Warrior 项目中怎么选？

既然你在处理 DataAsset 和 InputConfig，这里有个实战建议：

  1. 静态引用（Static Reference）：如果你在 Character 类的 UPROPERTY 里直接选好了 DataAsset，虚幻在角色实例化时通常会执行同步加载。如果 DataAsset 引用了太多重型资源（比如 100 个特效），你的游戏生成这个角色时就会闪卡一下。
  2. 软引用（Soft Object Reference）：如果你想做高性能优化，你会把变量类型改为 TSoftObjectPtr<UDataAsset>。这样它只存一个路径字符串，只有当你真正需要它时，才去手动调用 AsyncLoad。

本节战斧加载使用同步加载Synchronous

我们使用GiveToAbilitySystemComponent将能力与需要的ASC绑定好了，接下来只需要将角色与ASC绑定就搞定了，能力的实现在引擎里写好技能蓝图并写入DA_Hero即可
接下来要把DA_Hero这个数据资产绑定到角色上即可

  1. 在角色基类.h中声明如下
    CharacterStartUpData作为一个UDataAsset_StartUpDataBase数据资产的软引用，在玩家角色蓝图中，他将绑定DA_Hero，软引用需要使用.LoadSynchronous()手动加载

  2. 进入玩家角色用于初始化的PossessedBy()函数
    将之前的用于debug测试的输出尽数删除
    使用CharacterStartUpData.IsNull()进行判断是否指向了资源，无需关系资源中是否有东西，因为软引用不会直接加载资产
    使用CharacterStartUpData.LoadSynchronous()开始加载数据资产
    使用GiveToAbilitySystemComponent函数将数据资产里的技能授予给ASC，ASC里面就有东西了

  3. 在角色基类中的PossessBy()函数里添加一个ensureMsgf()进行对数据资产的检查，方便定位错误

    这样当我们开始运行时会后台输出报错

  4. 回到虚幻中的角色蓝图中配置配置好数据资产变量CharacterStartUpData
